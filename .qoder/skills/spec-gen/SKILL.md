# Spec 文档生成

## 适用场景

用户有一套 01~14 编号的 spec 模板，模板中含 `【填写】`、`【请填写】`、`【请补充】`、`【✓/✗】` 等占位标记，需要基于项目信息将它们填充为完整的规格文档。

## 文档体系

```
Proposal(03-04) → Spec(05-06-07) → Design(08-09-10-11) → Plan(12) → Test(13) → Trace(14)
  Why              What              How             When/Who     OK?      All linked?
```

| 编号 | 阶段 | 文档 | 回答什么 |
|------|------|------|---------|
| 03 | Proposal | 立项提案与范围说明 | 为什么做？做什么不做什么？ |
| 04 | Proposal | 产品需求说明 PRD | 做什么、多重要 |
| 05 | Spec | 用户故事与验收标准 | 谁要什么、怎么验收 |
| 06 | Spec | 功能规格说明 FSD | 界面怎么表现 |
| 07 | Spec | 非功能需求 NFR | 约束底线 |
| 08 | Design | 系统架构与技术选型 | 怎么组织、为什么 |
| 09 | Design | API 接口规格 | 后端必须返回什么（契约真源） |
| 10 | Design | 数据模型与存储规格 | 数据怎么存 |
| 11 | Design | 安全设计规格 | 安全怎么做 |
| 12 | Plan | 实施计划与里程碑 | 什么时候交 |
| 13 | Test | 测试策略与质量门禁 | 怎么验 |
| 14 | Trace | 需求追踪矩阵 | 全链路是否对齐 |

---

## 核心规则

### 写作约束（来自 01 总则）

1. **单一真相**：对外行为以 `09` API 与 `10` 数据模型为准
2. **先行为后实现**：先定义 `05` 用户故事，再写 `06/09/10`
3. **不混层**：PRD 不写 SQL，API 不写像素，Test 不重复规则
4. **API 字段名冻结在 09**：其他文档只引用不另起别名
5. **所有 AC 必须可映射到 TC-ID**

### 填充规则

- **只改占位符**：仅替换 `【填写】`、`【请填写】`、`【请补充】`、`【✓/✗】`、`【继续】` 等标记
- **保留已有内容**：已填写的文字、结构、示例一字不动
- **统一元数据**：模块编号、模块名称、日期在所有文件中保持一致
- **日期格式**：ISO-8601（如 2026-04-14）
- **推导而非编造**：所有填充内容必须从上游文档推导，标注来源

---

## 四轮螺旋写作顺序

> 编号是阅读顺序，写作按四轮推进（来自 01 §11）：

| 轮次 | 文档 | 做什么 | 清晰度 |
|------|------|--------|--------|
| **R1 起点** | 03 → 04 | 提炼原始素材 → 立项 → 产品需求 | 30% |
| **R2 锚定** | 05 + 09 | 用户故事与 API 接口互相校准 | 60% |
| **R3 展开** | 08 + 10 → 06 + 07 → 11 | 架构+数据先行，倒逼 FSD/NFR 补全，安全收尾 | 85% |
| **R4 收口** | 12 + 13 → 14 | 计划+测试并行；14 终检全链路 | 95% |

---

## R1 起点：03 立项提案 → 04 PRD

### 输入

- 需求采集记录（02）：会议纪要、邮件、用户访谈
- 项目背景信息

### 03 填充要点

| 章节 | 填什么 | 推导来源 |
|------|--------|---------|
| §1 背景与问题 | 2~3 条痛点 + 来源 | 02 采集记录 |
| §2 提案概要 | 3~5 句话概括做什么 | 02 需求归纳 |
| §3 范围边界 | 做什么（3~4 条）+ 不做什么（3~4 条） | 02 需求边界 |
| §4 成功标准 | 3~4 条可量化验收标准 | 痛点反转为正面目标 |
| §5 风险 | 2~3 个主要风险 + 影响 + 应对 | 技术依赖分析 |

### 04 填充要点

| 章节 | 填什么 | 推导来源 |
|------|--------|---------|
| §1 愿景目标 | 痛点 → 正面目标映射 | 03 §1 痛点 |
| §2 目标用户 | 角色 + 使用目标 + 频率 | 02 访谈对象 |
| §3 核心场景 | 4~5 条用户路径（SC-01~05） | 03 §2 提案 |
| §4 产品规则 | 输入约束、返回字段、降级策略等 | 03 §3 范围 + §4 标准 |
| §5 能力优先级 | P0/P1 排序 | 03 §3 范围 |

---

## R2 锚定：05 用户故事 + 09 API 互相校准

### 05 填充要点

- 每条 US 包含：需求编号、故事（角色+行为+价值）、优先级、验收标准 AC
- AC 必须可独立测试
- 每条 US 标注功能对齐（→06）、设计对齐（→08/09/10/11）、验证对齐（→13/14）

### 09 填充要点

- 每个端点写清：方法、路径、请求体字段表、成功响应字段表、错误响应
- 统一错误码体系：HTTP 状态码 + error.code + 触发条件 + details
- traceId 注入规则
- 参数校验规则汇总表

### 互相校准

```
05 的 AC 说"成功创建会话" → 09 必须有 POST /sessions → 201 的完整定义
09 的 POST /ask 需要 session_id → 05 的前置条件必须写"已创建会话"
```

---

## R3 展开：08 → 10 → 06 → 07 → 11

> 这是设计层填充的核心轮次。顺序很重要：架构先定，数据模型跟进，再倒逼功能和约束补全。

### 依赖链

```
08 架构（系统怎么组织）
  ↓
10 数据（数据怎么存）← 09 API 字段
  ↓
06 FSD（界面怎么表现）← 05 US/AC
  ↓
07 NFR（约束底线）← 03 风险 + 04 规则
  ↓
11 安全（安全怎么做）← 07 安全需求
```

### 08 系统架构 — 关键填充点

| 占位区域 | 填什么 | 推导来源 |
|---------|--------|---------|
| 架构决策表 | 选型 + 理由 | 03 技术栈 |
| 前端架构 · 样式方案 | 具体方案 + 理由 | 技术栈约束 |
| 存储演进路线 P1/P2 | 下一阶段 + 生产级存储 | 当前选型的自然演进 |
| ADR 决策记录 | "选 A 而非 B" + 核心理由 | 技术权衡 |
| 安全架构 | 当前措施 vs 生产改进 | 07 安全需求 |

### 10 数据模型 — 关键填充点

| 占位区域 | 填什么 | 推导来源 |
|---------|--------|---------|
| Storage 方法（update/delete） | 签名 + 行为 + 关联 TC | 09 API 端点行为 |
| 业务逻辑 | 触发条件 + 行为 | 05 AC 中隐含的规则 |

### 06 功能规格 — 关键填充点

| 占位区域 | 填什么 | 推导来源 |
|---------|--------|---------|
| 交互行为（选中会话等） | 触发 → API → 前端行为 | 05 US-001 |
| Main 三态渲染 | 各状态显示内容 | 05 US-002/003 |
| 错误码前端展示 | error.code → 用户提示文案 | 09 错误码 + 07 错误体系 |
| State 状态变量 | 变量名 + 类型 + 初始值 + 用途 | 05 AC 推导所需状态 |
| US → FSD 对齐表 | 每条 US 对应的本文章节 | 05 全部 US |

### 07 非功能需求 — 关键填充点

| 占位区域 | 填什么 | 推导来源 |
|---------|--------|---------|
| 延迟指标 | 各端点 SLO | 03 成功标准 |
| 故障恢复 | 恢复方式 | 08 存储架构 |
| 敏感数据保护 | 数据类型 + 保护方式 | 10 数据实体 |
| 输入约束（默认值等） | 字段默认值/规则 | 10 实体定义 |
| 类型约束 | 字段类型 + 校验规则 | 09 DTO 定义 |

### 11 安全设计 — 关键填充点

| 占位区域 | 填什么 | 推导来源 |
|---------|--------|---------|
| 安全目标第 N 条 | 目标 + 说明 + 合规依据 | 行业要求 |
| 认证机制 | 当前(教学版) + 生产建议 | 07 §4 安全需求 |
| 角色权限矩阵 | ✓/✗ 逐项填写 | 05 用户角色 + 02 访谈 |
| 加密策略 | 传输/存储/密钥 具体方案 | 07 敏感数据 |
| CSRF/验收标准 | 具体防护方式 / 通过标准 | 技术方案 |
| STRIDE 威胁 E/R | 威胁 + 等级 + 措施 + 负责人 | 安全分析 |
| 审计日志保留期限 | 具体天数 | 合规要求 |
| 部署安全补充 | 其他安全要求 | 运维最佳实践 |

---

## R4 收口：12 → 13 → 14

### 12 实施计划

- 按"可演示增量"切里程碑 S1~S4
- WBS 按工程职责切（路由/存储/Agent/前端/测试）
- REQ → S → W → TC 全链路对应表

### 13 测试策略

- TC 按"可独立失败的点"计数
- 4 层测试：Unit → Integration → Contract → E2E
- 质量门禁定义：G-LINT / G-UNIT / G-INT / G-AC / G-PERF

### 14 追踪矩阵

- 正向：REQ → US → API → TC
- 反向：TC → REQ
- API → TC 映射
- 覆盖率统计

---

## 质量门禁（文档侧，来自 01 §7）

填写完成后，检查以下条件：

- [ ] `05` 每条 P0 US 在 `14` 有映射
- [ ] `14` 每条 REQ 在 `13` 存在至少 1 个 TC
- [ ] `09` 每个端点有请求/响应/错误体示例
- [ ] `10` 实体有字段约束说明
- [ ] 所有文件无残留 `【填写】` / `【请填写】` / `【请补充】` 标记
- [ ] 模块编号/名称/日期全局一致
- [ ] API 字段名全局一致（以 09 为准）
